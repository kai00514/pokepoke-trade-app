### **📱**プロダクト概要：PokeLinkとは？

- **ポケポケ（Pokemon Pocket TCG）専用**のトレード掲示板サービス
- 「欲しいカード」と「譲れるカード」をスムーズに交換するためのプラットフォーム

**主要機能**

| 機能カテゴリ | 詳細 |
| --- | --- |
| **トレード掲示板** | 欲しいカード・譲れるカードを投稿してマッチング |
| **カードコラージュ生成** | カード一覧画像を自動生成してSNS共有 |
| **所有カードリスト管理** | 自分の所有カードをリスト化して管理 |
| **公式デッキガイド** | 管理者作成のデッキ構築ガイド |
| **デッキ共有・評価** | ユーザーが作成したデッキを共有・評価 |
| **コメント機能** | トレード投稿・デッキへのコメント |
| **通知システム** | コメント返信などの通知 |
| **ユーザー認証** | Supabase Auth（Google/GitHub OAuth） |
| **管理者機能** | 記事管理、大会情報管理、カードデータ管理 |

**対応言語（グローバル展開準備中）**

## **😰 やらかしと挫折**

### ① 「翻訳辞書作ればいいでしょ？」という甘い認識

まず言っておきたいのが、**グローバル展開を目指すなら初期設計の段階で考慮すべき**ということです。

当初の私の認識では、「UIのテキストを全部抽出して翻訳辞書作ればいいでしょ？ユーザー生成コンテンツは翻訳APIで対応すればいい」と思っていました。実際はそんな甘いものではなく、まず翻訳対象の「テキスト」を分類するところからスタートでした。

| コンテンツタイプ | 例 | 翻訳方法 | 難易度 |
| --- | --- | --- | --- |
| **UIテキスト** | ボタン、ラベル、エラーメッセージ | JSONファイル（next-intl） | ⭐ |
| **マスターデータ** | カード名「ピカチュウex」、パック名 | DBに事前格納 | ⭐⭐⭐ |
| **管理者コンテンツ** | デッキガイド、記事、大会情報 | DBに多言語カラム追加 | ⭐⭐⭐ |
| **ユーザー生成コンテンツ** | コメント、トレード投稿、デッキ説明 | リアルタイム翻訳API | ⭐⭐⭐⭐⭐ |

**解決策**:
- UIテキスト → Cursor上で全ページから網羅的に日本語テキストを抽出
- マスターデータ → 海外のポケポケサイト等からカード名やカード画像をスクレイピング
- 管理者コンテンツ → 日本語で作成した記事をGoogle Translateで対応言語に一括翻訳

**やらかし**: 翻訳作業は「UIテキストとユーザー生成コンテンツ」だと思い込んでいた。実際は4種類のコンテンツそれぞれに異なるアプローチが必要だった。そして、後述する翻訳コストも。。

---

### ② ユーザー生成コンテンツの翻訳コスト問題

**問題**:
- コメントやトレード投稿だけで**1ヶ月間で100万文字**にもなる（大誤算）
- Google Translateは200万文字を超えると課金対象
- フランス人ユーザーに日本語テキストをそのまま見せて「翻訳ボタンを押してね」というのはUX上微妙
- OSSの翻訳ライブラリはポケモンカード名などの固有名詞で変な結果になる

**解決策**:
- **翻訳機能と翻訳辞書機能の2つを持つ**必要があったため、主要な翻訳サービスの中で一番コストが抑えられるGoogle Translateを選定
- まずは翻訳ボタン方式を採用し、マネタイズができてから自動翻訳機能を実装する方針に

**やらかし**: コスト問題を甘く見ていた。なんならOSSで対応できると考えていた。翻訳APIの料金体系を事前に調査せず、「とりあえず実装してから考えよう」としていた。

---

### ③ カード名は翻訳APIでは正確に訳せない

**問題**:
```
入力: 「リザードンexが欲しいです」
Google Translateの結果: "I want a Lizard ex" ← ❌ 誤訳！
正しい結果: "I want a Charizard ex" ← ✅
```
ポケモンの名前は日本語と英語で全然違うものが多いので、翻訳APIに任せっきりにすると海外ユーザーには意味不明なテキストが表示されてしまう。

**解決策**:
- 2,259件のカード名すべてに対して7言語分の辞書を用意
- **カード名辞書を別途作成**して、翻訳後に置換する処理を実装

**やらかし**: 「翻訳APIに投げれば全部解決」と思っていた。固有名詞の扱いを完全に見落としていた。Google Translateの辞書機能があって本当に助かった。さすが天下のGoogle。

---

### ④ データベーススキーマの大改修

通常のプロジェクトでは、データベースのスキーマを変更するということは、サービス開発上それなりに大きな決断になります。

**問題**: 最初の設計が日本語専用だった
```sql
-- 日本語専用で設計していた
CREATE TABLE cards (
  id BIGINT PRIMARY KEY,
  name TEXT NOT NULL,  -- ← 日本語しか入らない
  image_url TEXT NOT NULL
);
```

**解決策**: 既存構造はそのままで、多言語対応カラムを追加
```sql
-- 13テーブルに48カラムのJSONB追加
ALTER TABLE cards ADD COLUMN name_multilingual JSONB;
-- {"ja": "ピカチュウex", "en": "Pikachu ex", "ko": "피카츄 ex", ...}

-- 26個のGINインデックス作成（検索性能のため）
CREATE INDEX idx_cards_name_multilingual ON cards USING GIN(name_multilingual);
```
バックエンド側のデータベースに挿入する処理も、このカラムに対して追加するように修正が必要でした。

**やらかし**: 完全移行できてから不要カラムは削除したいと思っていますが、消すという決断も難しい。ほぼ100%どこかしらから問題が起きて、移行できていない箇所を網羅的に調査して変更するのも面倒なので、現状維持も検討しています。

---

### ⑤ 既存データの量が想像以上だった

**問題**:
- サービスを運用していると、当然データは増え続ける
- トレード投稿だけで**149,000件以上**
- すべてに対して新しいJSONBカラムへのデータ移行が必要

**実際に必要だった作業量**:
- cards: 2,259件 → 7言語分のカード名をCSVからインポート
- trade_posts: 149,862件 → JSONBカラムへの変換
- deck_pages: 8件 → 17カラム×8件の翻訳
- その他多数...

**解決策**:
- マイグレーションスクリプトを作成して一括変換
- バッチ処理でDBへの負荷を分散

**やらかし**: 「あとでやればいい」と先延ばしにしていたツケが一気に回ってきた。データが少ないうちに対応していれば、こんな大変なことにはならなかった。ただこの辺りの作業はSupabaseが活躍しました。データベーストリガーを活用することで、テーブル間のリレーションを維持しながら、多言語対応カラムの追加と多言語データへの変換を同時に行うことができました。

### 📊 想定 vs 現実

| 項目 | 最初の想定 | 実際 |
|------|-----------|------|
| 作業期間 | 1週間 | 6週間以上（継続中） |
| 翻訳ファイル | 1〜2個 | 100ファイル以上 |
| DB変更 | 不要 | 13テーブル、48カラム追加 |
| スクリプト作成 | 不要 | 10個以上（マイグレーション、インポート等） |
| 外部API | 不要 | Google Cloud Translate（申請待ち発生） |
| コスト | 0円 | 翻訳API費用 + 開発工数 |

### 💡 教訓

1. **最初から多言語を見据えた設計をすべき**
   - スキーマは `name_multilingual JSONB` で設計
   - ハードコードされた日本語は最初から避ける

2. **「UIテキストだけ」という認識は甘い**
   - DBに保存されるコンテンツも多言語化対象
   - ユーザー生成コンテンツはリアルタイム翻訳が必要

3. **カード名などの固有名詞は翻訳APIでは正確に訳せない**
   - 辞書データの準備が必須

4. **既存データが増えるほど移行コストが増大**
   - 早めにやらないと149,000件の地獄を見る

## 🚀 今後の野望

### 目指すは海外M&Aで億万長者

正直に言うと、このサービスでグローバル展開を成功させて、最終的には海外企業へのM&A（売却）を狙っています。ポケモンは日本だけでなく海外ファンも非常に多いので、PokeLink海外版にニーズがあると考えています。

甘い期待かもしれませんが、理想としては**海外のポケモンファンである富裕層や、ゲーム系メディアを持っている企業への売却**です。自分で作ったサービスが、より大きなリソースを持つ会社の手に渡って、さらに成長していく姿を見届けられたら最高だなと思っています。

---

### マネタイズ戦略：30分マッチング機能

収益化の柱として考えているのが、「**30分以内にマッチングして欲しいカードの交渉ができる機能**」です。通常の掲示板だと、投稿して待つだけで相手が見つかるまで何時間もかかることがあります。この機能では、リアルタイムでマッチング相手を探し、すぐに交渉を開始できるようにします。利用料として課金することで、マネタイズを図る予定です。

ただし、この機能は**ユーザー数が一定以上いないと成立しません**。同時接続しているユーザーが少ないと、マッチング自体ができないからです。まずはユーザー基盤を固めることが先決です。

---

### 目標数値：ユーザー1万人、月売上10万円

漠然とした皮算用ですが、以下のような目標を立てています。

| 指標 | 目標値 |
|------|--------|
| アクティブユーザー数 | 3万人 |
| 月間売上 | 30万円 |
| 売却目標額 | 2,000万円 |

月売上10万円の安定したキャッシュフローがあれば、年間120万円。それを元に「安定収益のあるサービス」として売却交渉ができるのではないかと考えています。

---

### 売却前にやりたいこと：語学留学

せっかく安定したキャッシュが生まれるなら、**売却前に語学留学して英語をマスターしたい**とも考えています。自分で海外の交渉相手を見つけて、直接交渉するのも楽しそうだなと。仲介業者に任せるより、自分の言葉で価値を伝えられたほうが、より良い条件を引き出せる気がしています。

---

### リスク：ポケポケ公式のアップデート

一方で、大きなリスクも認識しています。それは**ポケポケ本体がトレード機能をアップデートする可能性**です。公式でトレード掲示板や交換機能が実装されたら、このサービスの存在意義は一気に薄れます。最悪の場合、サービス自体が破綻する可能性もあります。

だからこそ、**そうなる前にシナジーのある企業に売却して活用してもらう**のが理想です。ポケモン関連のメディアやコミュニティを持っている会社であれば、このサービスを彼らのエコシステムに組み込んでもらえる可能性があります。

---

### 売却後の展望：元手で起業

もしうまく売却できたら、その資金を元手に次の起業をしたいと考えています。ゼロからのスタートではなく、ある程度のリソースがある状態で始められれば、より大きなチャレンジができます。倍々で増やしていけるよう、努力を続けていきたいです。
